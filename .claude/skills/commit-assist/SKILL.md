# Commit Assist

Conventional Commitsスタイルのコミット文を生成し、ユーザーの選択に基づいてコミットを実行する。

## 重要な制約

- **Co-Authored-By は絶対に付けない**
- **git push は絶対に実行しない**
- コミット文は必ず Conventional Commits 形式にする

## ワークフロー

### Step 1: 変更分析

以下の2つの情報源からコミット内容を把握する:

1. **git差分の確認**: `git status` と `git diff`（staged + unstaged）を実行し、変更されたファイル・内容を把握する
2. **セッションコンテキストの活用**: このセッション内でユーザーと行った会話（何を実装したか、なぜ変更したか、どんな課題を解決したか）を振り返り、変更の意図・背景を把握する

両方を組み合わせて、変更の本質を理解する。

### Step 2: コミット文の生成

変更内容から以下の2種類のコミット文を生成する:

**簡潔版**: ヘッダー1行のみ
```
<type>(<scope>): <簡潔な説明>
```

**詳細版**: ヘッダー + 本文
```
<type>(<scope>): <簡潔な説明>

- 変更点1の説明
- 変更点2の説明
- 変更点3の説明
```

型・スコープの判定ルールは [references/conventional-commits.md](references/conventional-commits.md) を参照。

### Step 3: ユーザーへの提示と選択

以下の形式でユーザーに提示する:

```
## コミット文の候補

**簡潔版:**
feat(api): ユーザー認証エンドポイントを追加

**詳細版:**
feat(api): ユーザー認証エンドポイントを追加

- JWT認証ミドルウェアを実装
- /login, /logout エンドポイントを追加
- トークンリフレッシュ機能を実装
```

提示後、AskUserQuestion ツールを使って以下を1回の質問で確認する:

```
どちらのコミット文を使いますか？また、コミットするファイルを確認してください。

1. 簡潔版
2. 詳細版

コミット対象ファイル:
- src/auth/middleware.py
- src/auth/endpoints.py
- tests/test_auth.py

全てコミットする場合はそのまま番号を、除外したいファイルがあればファイル名も教えてください。
```

### Step 4: コミット実行

1. ユーザーが選択したファイルのみを `git add` でステージングする
2. 選択されたコミット文で `git commit` を実行する（HEREDOCを使用）
3. コミット結果を表示する
4. **push は実行しない**。必要ならユーザーが自分で行う旨を伝える

コミットコマンドの形式:
```bash
git add <file1> <file2> ...
git commit -m "$(cat <<'EOF'
<コミット文>
EOF
)"
```

## エラーハンドリング

### 変更がない場合
`git status` で変更がなければ、「コミットする変更がありません」と伝えて終了する。

### ステージング済みの変更がある場合
既にステージングされた変更がある場合は、その旨をユーザーに伝え、そのままコミットするか、追加の変更も含めるか確認する。

### コミット失敗時
pre-commit hook等でコミットが失敗した場合は、エラー内容を表示し、修正方法を提案する。再試行時は新しいコミットを作成する（--amend は使わない）。
