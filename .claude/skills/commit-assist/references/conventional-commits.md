# Conventional Commits 判定ルール

## 型（type）の判定

変更内容に基づいて以下から最も適切な型を選択する:

| 型 | 使用場面 |
|---|---|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `refactor` | 機能変更を伴わないコード改善 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響しない変更（フォーマット、セミコロン等） |
| `test` | テストの追加・修正 |
| `chore` | ビルドプロセス、依存関係、設定ファイル等の変更 |
| `perf` | パフォーマンス改善 |
| `ci` | CI/CD設定の変更 |
| `build` | ビルドシステム・外部依存関係の変更 |
| `revert` | 以前のコミットの取り消し |

### 判定の優先順位

1. 新しい機能が追加されている → `feat`
2. 既存のバグが修正されている → `fix`
3. 機能は変わらずコード構造が改善 → `refactor`
4. テストのみの変更 → `test`
5. ドキュメントのみの変更 → `docs`
6. 上記に当てはまらない → `chore`

## スコープ（scope）の判定

スコープは変更の影響範囲を示す。以下のルールで決定する:

1. **ディレクトリベース**: 変更が特定ディレクトリに集中 → そのディレクトリ名
   - 例: `src/api/` の変更 → `api`
2. **機能ベース**: 変更が特定機能に関連 → その機能名
   - 例: 認証関連の変更 → `auth`
3. **広範囲**: 変更が多くのファイルに分散 → スコープ省略可
   - 例: `refactor: コード全体のリンター対応`

スコープは短く（1-2語）、チーム内で一貫性のある命名にする。

## コミット文の言語

- プロジェクトの既存コミット履歴の言語に合わせる
- 日本語コミットが多ければ日本語、英語が多ければ英語で書く
- 判断がつかない場合は日本語で書く

## 簡潔版と詳細版の使い分け目安

- **簡潔版が適切**: 単一の小さな変更、1-2ファイルの修正、意図が明確な変更
- **詳細版が適切**: 複数ファイルにまたがる変更、複数の関連する修正、背景説明が必要な変更
