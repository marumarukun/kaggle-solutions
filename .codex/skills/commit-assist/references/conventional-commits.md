# Conventional Commits判定ルール

## 形式

```text
<type>[optional scope][!]: <description>

[optional body]

[optional footer]
```

- descriptionは簡潔な命令・要約として書き、末尾に句点を付けない。
- 破壊的変更ではtypeまたはscopeの直後に `!` を付け、必要なら `BREAKING CHANGE:` footerで影響と移行方法を説明する。
- issue番号などのfooterは、リポジトリの慣例またはユーザーの指定がある場合だけ加える。

## typeの選択

| type | 使用場面 |
|---|---|
| `feat` | 利用者から見える新機能 |
| `fix` | バグ修正 |
| `refactor` | 動作変更を目的としないコード構造の改善 |
| `docs` | ドキュメントだけの変更 |
| `style` | 意味に影響しない書式変更 |
| `test` | テストだけの追加・修正 |
| `chore` | 保守作業や設定変更 |
| `perf` | パフォーマンス改善 |
| `ci` | CI/CD設定の変更 |
| `build` | ビルドシステムや依存関係の変更 |
| `revert` | 以前のコミットの取り消し |

変更が複数typeにまたがる場合は、コミットの主目的を選ぶ。独立して説明できる変更なら、1件に詰め込まず分割を提案する。

## scopeの選択

scopeは任意とし、リポジトリの既存履歴を優先する。

1. 変更が特定コンポーネントに集中する場合は、その短い名称を使う。
2. ディレクトリ名より機能名のほうが意図を表す場合は、機能名を使う。
3. 広範囲の変更や既存履歴がscopeを使わない場合は省略する。
4. 新しいscopeを作る前に `git log` で既存の綴りを確認する。

## 言語

- 既存コミット履歴の主な言語に合わせる。
- ユーザーが言語を指定した場合は、その指定を優先する。
- 判断できない場合は、ユーザーとの会話で使われている言語を使う。

## 簡潔版と詳細版

- 単一の小さな変更で意図が明確なら、ヘッダーだけの簡潔版を優先する。
- 複数ファイル、複数の関連修正、背景や移行上の注意がある場合は本文付きの詳細版を優先する。
- 本文は「何を変えたか」「なぜ必要か」「利用者や運用への影響」のうち、判断に必要な点だけを書く。
