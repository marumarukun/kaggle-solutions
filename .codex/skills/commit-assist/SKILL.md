# Commit Assist

変更の内容と意図を把握し、安全にスコープを確定してからコミットする。

## 守ること

- コミット文を Conventional Commits 形式にする。
- `Co-Authored-By` を付けない。
- `git push` を実行しない。
- ユーザーが明示的に依頼しない限り `--amend` を使わない。
- ユーザーの変更を `reset`、`checkout`、`restore` などで破棄しない。
- 選択されていない変更、秘密情報、認証情報をコミットしない。
- 候補を提示する段階ではGitの状態を変更しない。

## 1. 変更を分析する

次を確認する。

```bash
git status --short
git diff --stat
git diff
git diff --cached --stat
git diff --cached
git log -10 --format=%s
```

- staged、unstaged、untrackedを区別する。
- untrackedファイルは差分に出ないため、コミット候補なら内容を個別に確認する。
- セッション内で実施した作業から、変更の目的と背景を補う。
- 既存コミット履歴から、コミット文の主な言語とscopeの慣例を判断する。
- 変更がなければ「コミットする変更がありません」と伝えて終了する。

## 2. コミット候補を作る

候補作成前に [references/conventional-commits.md](references/conventional-commits.md) を読む。

原則として次の2案を作る。

```text
簡潔版:
<type>(<scope>): <要約>

詳細版:
<type>(<scope>): <要約>

- 重要な変更点
- 変更理由または影響
```

- 小さな単一変更では詳細版を無理に長くしない。
- 本文には差分から確認できる事実と、セッションで確認できた意図だけを書く。
- 複数の独立した変更が混在する場合は、分割コミット案を提示する。
- ユーザーが既に簡潔版・詳細版、文面、対象ファイルを指定していれば、不要な再確認をしない。

選択が必要な場合は、最終回答に以下をまとめて1回だけ尋ねる。

- 簡潔版と詳細版の全文
- 各案でコミットするファイル
- 既にstagedされているファイル
- 「簡潔版」「詳細版」または修正文面と、除外するファイルを答えてもらう短い質問

## 3. 選択された変更をステージする

ユーザーの回答後に `git status --short` を再確認する。状態が変わっていれば差分を再分析し、対象が変わる場合だけ確認する。

- ユーザーが選んだファイルだけを `git add -- <paths>` で追加する。
- 「ステージ済みだけ」と指定された場合は、unstagedの内容を追加しない。
- 同じファイルにstagedとunstagedの両方がある場合、依頼された側だけを扱う。
- 対象外のファイルが既にstagedされている場合は勝手にunstageせず、含めるかどうか確認して停止する。

コミット直前に次を実行し、対象と内容を確認する。

```bash
git diff --cached --check
git diff --cached --stat
git diff --cached
```

秘密鍵、トークン、`.env`、資格情報らしい値、大きな意図しないバイナリが見つかった場合はコミットせず、該当箇所をユーザーへ伝える。

## 4. コミットする

ヘッダーを最初の `-m`、本文を2つ目の `-m` に渡す。HEREDOCやコマンド置換は使わず、各引数を安全にshell quoteする。

```bash
git commit -m '<type>(<scope>): <要約>' -m '<本文>'
```

簡潔版では本文用の2つ目の `-m` を省略する。コミット後は次を確認する。

```bash
git show --stat --oneline --summary HEAD
git status --short
```

コミットハッシュ、件名、残っている変更を報告する。pushは行わない。

## エラー処理

- pre-commit hookなどで失敗したら、エラーと現在のGit状態を示す。
- hookがファイルを変更した場合は差分を再確認し、勝手に再ステージしない。
- 再試行では新しいコミットを作り、ユーザーが明示しない限り `--amend` しない。
- コンフリクト、merge、rebaseの途中なら通常のコミットとして進めず、状態と必要な判断を説明する。

## 提示例

```text
コミット文の候補

簡潔版:
feat(auth): JWT認証を追加

詳細版:
feat(auth): JWT認証を追加

- ログインとログアウトのエンドポイントを実装
- トークン更新処理と認証テストを追加

対象ファイル:
- src/auth.py
- tests/test_auth.py

簡潔版と詳細版のどちらでコミットしますか？除外するファイルがあれば併せて教えてください。
```
